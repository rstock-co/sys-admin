## ✨ v2.1.50 - February 21, 2026

### What Changed
Added `startupTimeout` configuration for LSP servers. Added `WorktreeCreate` and `WorktreeRemove` hook events for custom VCS setup/teardown when agent worktree isolation creates or removes worktrees. Added `claude agents` CLI command to list configured agents. Opus 4.6 (fast mode) now includes full 1M context window. Added `CLAUDE_CODE_DISABLE_1M_CONTEXT` environment variable. Enhanced `CLAUDE_CODE_SIMPLE` to strip down skills, session memory, custom agents, CLAUDE.md token counting, MCP tools, attachments, hooks, and CLAUDE.md file loading. Fixed resumed sessions becoming invisible when working directory involved symlinks. Fixed native modules failing on systems with older glibc (RHEL 8). Fixed `/mcp reconnect` freezing with nonexistent server name. Fixed MCP tool discovery when tool search is enabled with a prompt launch argument. Fixed prompt suggestion cache regression. Multiple memory leak fixes: agent teams completed teammate tasks, task state objects in AppState, LSP diagnostic data, completed task output, TaskOutput retained lines, CircularBuffer cleared items, shell ChildProcess references, file history snapshots (now capped). Improved memory usage by clearing internal caches and large tool results after processing. Improved headless mode startup by deferring Yoga WASM and UI imports. VSCode: `/extra-usage` command support.

### What It Means
**Worktree Hook Events:**
Building on v2.1.49's `--worktree` flag, new `WorktreeCreate` and `WorktreeRemove` hooks fire when worktrees are created or destroyed. Use cases:
- Custom VCS setup (install hooks, configure git settings in new worktrees)
- Cleanup operations when worktrees are removed
- Logging worktree lifecycle for debugging parallel agent work

**Opus 4.6 1M Context Window:**
Fast mode (Opus 4.6) now gets the full 1M token context window. Previously 1M was limited - now all Opus 4.6 users benefit. Use `CLAUDE_CODE_DISABLE_1M_CONTEXT` to opt out if needed.

**`claude agents` CLI:**
New CLI command to list all configured agents. Useful for verifying agent configurations without reading files.

**Massive Memory Leak Cleanup:**
This release fixes at least 8 distinct memory leaks affecting agent teams, task management, LSP, shell execution, and file history. Critical for long-running sessions and parallel agent work.

## 🚀 v2.1.49 - February 20, 2026

### What Changed
Added `--worktree` (`-w`) flag to start Claude in an isolated git worktree. Subagents support `isolation: "worktree"` for temporary git worktrees. Added Ctrl+F keybinding to kill background agents (two-press confirmation). Agent definitions support `background: true` to always run as background tasks. Plugins can ship `settings.json` for default configuration. Added `ConfigChange` hook event that fires when configuration files change during session. Sonnet 4.5 with 1M context being removed in favor of Sonnet 4.6. SDK model info includes `supportsEffort`, `supportedEffortLevels`, and `supportsAdaptiveThinking` fields. Fixed file-not-found errors to suggest corrected paths. Fixed Ctrl+C and ESC being silently ignored when background agents running. Fixed `plugin enable`/`plugin disable` scope auto-detection. Fixed `disableAllHooks` to respect managed settings hierarchy. Fixed `--resume` session picker showing raw XML tags. Fixed verbose mode not updating thinking block display. Fixed unbounded WASM memory growth by periodically resetting tree-sitter parser. Fixed unbounded Yoga WASM linear memory. Simple mode now includes file edit tool. Permission suggestions now populated when safety checks trigger. Improved permission prompts to show reason for restrictions. Improved startup performance by caching MCP auth failures, reducing HTTP calls for analytics, and batching MCP tool token counting.

### What It Means
**`--worktree` Flag:**
Start Claude Code in its own isolated git worktree:
```bash
claude --worktree        # Auto-named worktree
claude -w my-feature     # Named worktree
claude --worktree --tmux # In its own tmux session
```
Each worktree gets an independent copy of the repo. Multiple Claude sessions can work on different branches simultaneously without code edits conflicting. This is a paradigm shift for parallel work - agents can modify files independently.

Subagents can also use worktree isolation:
```javascript
// In agent definitions
{ "isolation": "worktree" }
```

**Agent `background: true`:**
Agent definitions can declare `background: true` to always run as background tasks. No manual Ctrl+B needed - spawn it and it runs in the background automatically.

**Ctrl+F Kill Background Agents:**
Two-press confirmation (Ctrl+F, Ctrl+F) to kill all background agents. Clean shutdown for runaway processes.

**ConfigChange Hook:**
New hook event that fires when configuration files change during a session. Use cases:
- Detect when `.claude/rules/` files are modified
- Audit configuration changes for security
- React to settings changes in real-time

**Sonnet 4.5 → 4.6 Transition:**
Sonnet 4.5 with 1M context is being removed. Sonnet 4.6 replaces it as the mid-tier model.

### How to Use It
```bash
# Start in isolated worktree
claude --worktree

# With tmux for persistent sessions
claude --worktree --tmux

# Named worktree
claude -w feature-branch
```

## 🚀 v2.1.32 - February 5, 2026

### What Changed
Claude Opus 4.6 model release. Research preview agent teams for multi-agent collaboration (requires `CLAUDE_CODE_EXPERIMENTAL_AGENT_TEAMS=1`). Automatic memory recording and recall functionality. "Summarize from here" for partial conversation summarization. Skills from `--add-dir` directories loading. @-mention path completion fix for subdirectories. Updated `--resume` to re-use `--agent` values from previous sessions. Bash heredoc template literal fix. Skill character budget scaling with context window. Thai/Lao character rendering improvements. VSCode slash command and conversation loading fixes.

### What It Means
**Claude Opus 4.6:**
Anthropic's newest flagship model, released February 5, 2026 - roughly three months after Opus 4.5. This is the model powering this conversation.

*Key Improvements:*
- **Adaptive Thinking** - Claude dynamically decides when and how much to think (replaces manual extended thinking toggle). At default effort level (high), Claude will almost always think.
- **1M Token Context Window (Beta)** - A first for Opus-class models. 5x the previous 200k limit.
- **128K Max Output Tokens** - Double the previous 64K limit.
- **Stronger Agentic Coding** - Best SWE-bench scores, better planning, code review, and debugging. Catches its own mistakes more reliably.
- **Large Codebase Navigation** - More reliable in large codebases, sustains agentic tasks for longer.
- **Better Document Analysis** - Improved at pulling relevant information from large document sets, research, and financial analysis.
- **Same Pricing** - $5/$25 per million tokens (unchanged from Opus 4.5).

*Effort Parameter:*
Opus 4.6 introduces the `effort` parameter to control reasoning depth:
- `high` (default) - Deep reasoning, almost always thinks
- `medium` - Balanced speed/quality
- `low` - Quick responses, minimal thinking

**Agent Teams (Research Preview):**
The most significant new primitive since background agents. Agent teams enable multiple Claude Code instances to coordinate autonomously:

- One session acts as **team lead**, coordinating work and synthesizing results
- **Teammates** work independently, each in its own context window
- Teammates can **challenge each other's findings** and communicate directly
- Uses **tmux** for display - each teammate gets its own pane

Enable with:
```bash
export CLAUDE_CODE_EXPERIMENTAL_AGENT_TEAMS=1
```

Best for tasks with clear boundaries: reviewing PRs, researching libraries, investigating bugs, parallel feature implementation.

**Automatic Memory:**
Claude Code now automatically records and recalls memories as it works:
- Records to `~/.claude/projects/{project-path}/memory/MEMORY.md`
- First extraction after ~10,000 tokens of conversation
- Subsequent updates every ~5,000 tokens or 3 tool calls
- Session summaries stored as structured markdown
- No manual intervention needed - builds project history automatically

**Summarize from Here:**
Partially compact a conversation from a specific point, rather than compacting everything. Useful for preserving early context while freeing up recent verbose output.

**Skills from `--add-dir`:**
Skills/commands from directories added via `--add-dir` now load properly. Combined with v2.1.20's CLAUDE.md loading from `--add-dir`, this completes the picture - both instructions AND skills can be shared across sessions.

### How to Use It
```bash
/model opus                     # Switch to Opus 4.6
claude --model opus             # Start with Opus 4.6
/fast                           # Toggle fast mode (same model, faster output)

# Agent Teams (experimental)
export CLAUDE_CODE_EXPERIMENTAL_AGENT_TEAMS=1
claude                          # Teams enabled in session
```

## ✨ v2.1.30 - February 3, 2026

### What Changed
Read tool `pages` parameter for PDF page ranges. Pre-configured OAuth for MCP servers without DCR support. `/debug` command for session troubleshooting. Additional git log/show flags support. Task tool metrics (token count, duration). Reduced motion mode option. Prompt cache invalidation improvement. 68% session resume memory optimization. VSCode multiline input support.

### What It Means
**PDF Page Ranges:**
The Read tool now accepts a `pages` parameter:
```
Read("document.pdf", pages: "1-5")    # Read pages 1-5
Read("document.pdf", pages: "10-20")  # Read pages 10-20
```
Essential for large PDFs where you only need specific sections. Maximum 20 pages per request.

**/debug Command:**
New troubleshooting command for session issues:
```bash
/debug    # Show session diagnostics
```
Helps identify configuration problems, MCP server issues, and other session-level problems.

**68% Session Resume Memory Optimization:**
Resuming sessions now uses 68% less memory. Faster resume times, especially for sessions with large conversation histories.

**Task Tool Metrics:**
Task tool calls now include token count and duration metrics. Track how much each task costs and how long it takes.
