## Summary

v2.0.64 is a landmark release introducing **asynchronous agent execution**, **named sessions**, and the **`.claude/rules/` directory** - three foundational features for autonomous operation. Async agents enable parallel workflows where sub-agents work in the background while you continue with the main session. Named sessions provide persistent, resumable workflows. The rules directory enables modular, path-targeted project configuration.

## What Changed

### New Features
- **Asynchronous Execution** - Agents and bash commands can run in background, sending wake-up notifications to main agent
- **Named Sessions** - Use `/rename` to name sessions, resume by name with `claude --resume session-name`
- **`.claude/rules/` Directory** - Modular project configuration with automatic loading and path targeting
- **Auto-compacting** - Compaction process is now instantaneous
- **`/stats` Command** - Personalized insights: favorite model, usage graph, usage streak
- **Image Metadata** - Dimension metadata included when images are resized for coordinate mapping

### Bug Fixes & Improvements
- Fixed `.env` auto-loading with native installer
- Fixed `--system-prompt` being ignored with `--continue` or `--resume` flags
- Enhanced `/resume` interface with grouped forked sessions and keyboard shortcuts (P preview, R rename)
- VSCode: Copy-to-clipboard on code blocks and bash tool inputs
- VSCode: Fixed Windows ARM64 support via x64 emulation
- Bedrock: Streamlined token counting efficiency

## Deep Dive: Asynchronous Agent Execution

### The Problem It Solves

Before v2.0.64, sub-agents blocked the main conversation. If Claude spawned a research agent that needed to read 50 files, you waited. Complex workflows couldn't parallelize - everything was sequential.

### How Ctrl+B Backgrounding Works

1. **Claude spawns a sub-agent** - Main agent delegates work to specialized sub-agent
2. **User presses Ctrl+B** - While agent is actively running (not after completion)
3. **Agent moves to background** - Execution continues independently
4. **User continues main session** - Chat with main agent on unrelated tasks
5. **Sub-agent completes** - Wake-up notification surfaces results to main agent

**Critical timing**: Press Ctrl+B while the agent is actively running. Not before (doesn't start), not after (already done).

**tmux users**: Use Ctrl+B Ctrl+B (double press) since Ctrl+B is tmux's prefix key.

### Agent Communication Architecture

| Component | Role |
|-----------|------|
| Main Agent | Orchestrates, spawns sub-agents, receives results |
| Background Agent | Executes independently, sends wake-up notifications |
| `/tasks` Command | Shows all running background processes with IDs |

Background agents:
- Inherit parent's permissions
- Auto-deny anything not pre-approved
- Run in separate shells with unique IDs (bash_1, bash_2, etc.)
- Can notify main agent when they need attention
- Don't require polling - system handles notification

### Optimal Use Cases

**Background-friendly tasks:**
- Web research requiring multiple searches
- Large codebase analysis (grep/read many files)
- Documentation generation
- Security vulnerability scanning
- Performance profiling
- Test suite execution

**Foreground-required tasks:**
- Decisions needing immediate user input
- Code modifications requiring review
- Work with sequential dependencies
- Permission-sensitive operations

### Managing Background Agents

```bash
# View all background tasks
/tasks

# Output shows:
# bash_1: Running npm install... (15s)
# agent_1: Analyzing codebase... (45s)
# bash_2: Completed (exit 0)
```

Each task has a unique ID for monitoring status, viewing output, or terminating.

## Deep Dive: Named Sessions

### The Problem It Solves

Before v2.0.64, resuming sessions required remembering cryptic session IDs or scrolling through a list of "explain this function" style descriptions. Managing multiple parallel workstreams was friction-heavy.

### How Named Sessions Work

**Naming a session:**
```bash
# From within Claude Code
/rename auth-refactor
/rename payment-debugging
/rename health-coach-review
```

**Resuming by name:**
```bash
# From terminal
claude --resume auth-refactor
claude --resume payment-debugging --model opus

# From within Claude Code
/resume auth-refactor
```

### The `/resume` Interface

v2.0.64 enhanced the resume screen with:

| Shortcut | Action |
|----------|--------|
| `P` | Preview conversation history |
| `R` | Rename the selected session |
| `Enter` | Resume the session |

**Forked sessions** now group together visually, making branching experiments easier to track.

### Session Storage

Sessions persist in a local SQLite database:
- `$CLAUDE_CONFIG_DIR/.claude/sessions.db` (if set)
- `~/.claude/sessions.db` (default)
- `$XDG_CONFIG_HOME/claude/sessions.db` (Linux with XDG)

Named sessions survive restarts - name once, resume indefinitely.

### Best Practices

- **Name early**: Use `/rename` when starting work on a distinct task
- **Be descriptive**: `payment-integration` > `explain this function`
- **Use for parallel workstreams**: Different sessions for different features
- **Session forking**: Branch experiments from working sessions

## Deep Dive: `.claude/rules/` Directory

### The Problem It Solves

Large projects with complex CLAUDE.md files suffer from:
1. **Monolithic maintenance** - One 500+ line file becomes unwieldy
2. **Priority saturation** - Everything competes equally for attention
3. **No conditional loading** - API rules load even when working on UI

### How `.claude/rules/` Works

```
.claude/rules/
├── code-style.md        # Universal - no frontmatter
├── security.md          # Conditional - paths: src/auth/**, src/payments/**
├── testing.md           # Conditional - paths: **/*.test.ts
└── frontend/
    ├── react.md         # Conditional - paths: src/components/**
    └── styles.md        # Conditional - paths: **/*.css
```

**Automatic loading:**
- All `.md` files in `.claude/rules/` auto-load
- Same priority as `CLAUDE.md`
- No registration or configuration required
- Subdirectories supported

### Path-Specific Targeting

Rules can conditionally activate using YAML frontmatter:

```yaml
---
paths: src/api/**/*.ts
---
# API Development Rules
These rules only load when working on TypeScript files in src/api/
```

**Pattern syntax:**

| Pattern | Matches |
|---------|---------|
| `src/api/**/*.ts` | All .ts files in src/api recursively |
| `*.test.ts` | Test files in any directory |
| `src/components/*.tsx` | Direct children only (not nested) |
| `**/*.css` | CSS files anywhere in project |

**Multiple paths:**
```yaml
---
paths:
  - src/components/**/*.tsx
  - src/hooks/**/*.ts
---
```

**No paths = always loaded** - Rules without frontmatter behave like CLAUDE.md content.

### User-Level Rules

Create personal rules at `~/.claude/rules/` that apply across all projects:
- User rules load first (lower priority)
- Project rules override user rules
- Good for personal coding style, preferred tools

### Symlink Support

Rules directory supports symlinks:
- Share common rules across projects
- Link to team-standard rules
- Circular symlinks detected and handled gracefully

### When to Use Rules vs CLAUDE.md

| Use Case | Where |
|----------|-------|
| Routing logic | CLAUDE.md |
| Quality standards | CLAUDE.md |
| Coordination protocols | CLAUDE.md |
| API patterns | rules/ with paths targeting |
| Test requirements | rules/ with *.test.* targeting |
| Security constraints | rules/ with auth/payments targeting |

## Symbiont Impact

### Foundation for Autonomous Operation

v2.0.64's features combine to enable sophisticated autonomous workflows:

**1. Parallel Agent Execution**

Before v2.0.64: Single-threaded - one agent does everything sequentially.

After v2.0.64:
```
Main Agent: "Let me analyze this codebase..."
├── [Background] Research Agent: Scanning 200 files
├── [Background] Security Agent: Checking vulnerabilities
├── [Foreground] Main Agent: Discussing architecture with user
│
└── All results merge when background agents complete
```

**2. Persistent Workstreams**

Named sessions enable long-running autonomous contexts:

```bash
# Morning: Start health analysis
claude --resume health-daily

# Background: Run overnight processing
claude --resume overnight-daemon

# Afternoon: Continue where you left off
claude --resume health-daily
```

Each specialist gets a named, persistent context that survives restarts.

**3. Modular Specialist Configuration**

`.claude/rules/` enables:
- Per-specialist rules in their working directories
- Shared core rules via symlinks
- Path-targeted activation for context-appropriate behavior

### Integration Pattern for SGTAs

```
mycelium/health-coach/
├── CLAUDE.md                 # Core persona
├── .claude/
│   └── rules/
│       ├── health-data.md    # paths: **/*.health.json
│       ├── recommendations.md # paths: **/recommendations/**
│       └── core/ → symlink   # Shared SGTA rules
```

**Spawning pattern:**
```bash
# Named session for persistent context
claude --working-directory ~/symbiont/mycelium/health-coach \
       --resume health-coach-main
```

### Parallel Specialist Workflows

With async agents, Mycelia could:

1. Spawn Health Coach (backgrounded)
2. Spawn Business Savant (backgrounded)
3. Continue orchestration work
4. Receive results from both when complete

```
Mycelia Session
├── [Background] Health Coach: Analyzing Oura data
├── [Background] Business Savant: Reviewing financial metrics
├── [Foreground] Mycelia: Coordinating synthesis
│
└── Wake-up notifications when specialists complete
```

### Action Items

1. **Name specialist sessions** - Each SGTA gets a persistent named session
2. **Use rules/ for path-targeted config** - Reduce context pollution
3. **Experiment with parallel agents** - Test backgrounding for research tasks
4. **Symlink shared rules** - core/ rules accessible to all specialists
5. **Test wake-up notifications** - Verify background completion surfacing

---

## Limitations

1. **Ctrl+B timing** - Must press while agent is running, not after completion
2. **Permission inheritance** - Background agents auto-deny non-pre-approved permissions
3. **Session-local** - Named sessions don't sync across machines
4. **Rules loading** - Very large rules directories may impact startup time
