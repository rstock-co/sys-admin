# Claude Code v2.0.45 - Deep Research

**Release Date:** November 19, 2025
**Tier:** Major
**Symbiont Relevance:** High
**Researched:** 2026-01-27

---

## Summary

v2.0.45 introduces the **PermissionRequest hook** - a foundational primitive for autonomous operation. This hook fires when a permission dialog appears and allows programmatic auto-approval or denial of tool permission requests. Combined with `PreToolUse` hooks (existing), you now have two layers of permission interception: one before tool creation, one at permission prompt time. This release also adds Azure AI Foundry support and background tasks on web (`&` prefix).

---

## What Changed

- **PermissionRequest hook**: Automatically approve or deny tool permission requests with custom logic
- **Azure AI Foundry support**: Enterprise integration with Microsoft's AI infrastructure
- **Background tasks on web**: Start messages with `&` to run tasks asynchronously on claude.ai/code
- **Permission hook processing**: Later enhanced to process "always allow" suggestions and apply permission updates

## Deep Dive: PermissionRequest Hook

### The Problem It Solves

Before v2.0.45, permission handling had gaps:

| Layer | Timing | Capabilities |
|-------|--------|--------------|
| **PreToolUse** | Before tool call is created | Allow, deny, or ask - but tool hasn't been fully prepared yet |
| **Permission Dialog** | At user prompt time | Manual approval required - interrupts autonomous operation |

The gap: Even with PreToolUse hooks allowing a tool, Claude Code might still show a permission dialog based on permission mode settings. There was no way to programmatically respond to these dialogs.

### How PermissionRequest Hook Works

```
Tool Call Flow:

  Claude creates tool call
         │
         ▼
  ┌─────────────────┐
  │  PreToolUse     │──▶ Can deny/allow/ask before tool prepared
  └─────────────────┘
         │
         ▼
  ┌─────────────────┐
  │  Deny Rules     │──▶ settings.json deny patterns
  │  Allow Rules    │──▶ settings.json allow patterns
  │  Ask Rules      │──▶ settings.json ask patterns
  │  Permission Mode│──▶ default, plan, acceptEdits, etc.
  └─────────────────┘
         │
         ▼
  Permission dialog would show
         │
         ▼
  ┌─────────────────────────┐
  │  PermissionRequest Hook │──▶ NEW: Programmatic auto-approve/deny
  └─────────────────────────┘
         │
         ▼
  Tool executes (or denied)
```

### Input Schema

The PermissionRequest hook receives JSON via stdin:

```json
{
  "session_id": "abc123",
  "transcript_path": "/home/neo/.claude/projects/.../transcript.jsonl",
  "cwd": "/home/neo/projects/myproject",
  "permission_mode": "default",
  "hook_event_name": "PermissionRequest",
  "tool_name": "Bash",
  "tool_input": {
    "command": "npm install",
    "description": "Install dependencies",
    "timeout": 120000
  },
  "tool_use_id": "toolu_01ABC123..."
}
```

| Field | Description |
|-------|-------------|
| `session_id` | Unique session identifier |
| `transcript_path` | Path to session transcript JSONL |
| `cwd` | Current working directory |
| `permission_mode` | Current mode: "default", "plan", "acceptEdits", "dontAsk", "bypassPermissions" |
| `hook_event_name` | Always "PermissionRequest" |
| `tool_name` | Tool requesting permission (e.g., "Bash", "Write", "mcp__memory__create") |
| `tool_input` | The tool's input parameters |
| `tool_use_id` | Unique identifier for this tool call |

### Output Schema: Allow

To auto-approve a permission request:

```json
{
  "hookSpecificOutput": {
    "hookEventName": "PermissionRequest",
    "decision": {
      "behavior": "allow"
    }
  }
}
```

You can also modify the tool input before approval:

```json
{
  "hookSpecificOutput": {
    "hookEventName": "PermissionRequest",
    "decision": {
      "behavior": "allow",
      "updatedInput": {
        "command": "npm run lint --fix"
      }
    }
  }
}
```

### Output Schema: Deny

To programmatically deny a permission request:

```json
{
  "hookSpecificOutput": {
    "hookEventName": "PermissionRequest",
    "decision": {
      "behavior": "deny",
      "message": "Production database commands are blocked",
      "interrupt": true
    }
  }
}
```

| Field | Description |
|-------|-------------|
| `behavior` | "deny" to reject the request |
| `message` | Explanation shown to Claude (informs its next action) |
| `interrupt` | If `true`, stops Claude entirely; if `false`, Claude continues |

### Output Schema: Prompt User (Passthrough)

To let the permission dialog show normally, output nothing or:

```json
{
  "hookSpecificOutput": {
    "hookEventName": "PermissionRequest",
    "decision": {
      "behavior": "prompt"
    }
  }
}
```

## Configuration Examples

### Basic PermissionRequest Hook

Add to `~/.claude/settings.json`:

```json
{
  "hooks": {
    "PermissionRequest": [
      {
        "matcher": "Bash",
        "hooks": [
          {
            "type": "command",
            "command": "/home/neo/.claude/hooks/bash-permission.sh"
          }
        ]
      }
    ]
  }
}
```

### Auto-Approve Safe MCP Tools

Pattern-based approval for read-only operations:

```javascript
#!/usr/bin/env node
// ~/.claude/hooks/mcp-permission.js

const SAFE_PATTERNS = [
  'get', 'list', 'read', 'retrieve', 'fetch', 'search',
  'find', 'query', 'recall', 'check', 'status', 'stats'
];

const DESTRUCTIVE_PATTERNS = [
  'delete', 'remove', 'destroy', 'drop', 'clear', 'wipe',
  'update', 'modify', 'write', 'create', 'execute'
];

async function main() {
  const input = JSON.parse(await readStdin());
  const toolName = input.tool_name.split('__').pop(); // Strip mcp__ prefix

  // Check destructive first (deny)
  if (DESTRUCTIVE_PATTERNS.some(p => toolName.includes(p))) {
    // Let user decide
    console.log(JSON.stringify({
      hookSpecificOutput: {
        hookEventName: "PermissionRequest",
        decision: { behavior: "prompt" }
      }
    }));
    return;
  }

  // Auto-approve safe operations
  if (SAFE_PATTERNS.some(p => toolName.includes(p))) {
    console.log(JSON.stringify({
      hookSpecificOutput: {
        hookEventName: "PermissionRequest",
        decision: { behavior: "allow" }
      }
    }));
    return;
  }

  // Unknown - prompt user
  console.log(JSON.stringify({
    hookSpecificOutput: {
      hookEventName: "PermissionRequest",
      decision: { behavior: "prompt" }
    }
  }));
}

function readStdin() {
  return new Promise(resolve => {
    let data = '';
    process.stdin.on('data', chunk => data += chunk);
    process.stdin.on('end', () => resolve(data));
  });
}

main();
```

### Bash Command Whitelist

Auto-approve specific commands, deny dangerous ones:

```bash
#!/bin/bash
# ~/.claude/hooks/bash-permission.sh

input=$(cat)
command=$(echo "$input" | jq -r '.tool_input.command')

# Extract first word (handling VAR=value prefixes)
first_word=$(echo "$command" | awk '{for(i=1;i<=NF;i++){if(index($i,"=")==0){print $i;exit}}}')

# Deny dangerous commands
case "$command" in
  *"rm -rf /"*|*"rm -rf ~"*|*"sudo"*)
    echo '{"hookSpecificOutput":{"hookEventName":"PermissionRequest","decision":{"behavior":"deny","message":"Dangerous command blocked","interrupt":true}}}'
    exit 0
    ;;
esac

# Allow safe read-only commands
case "$first_word" in
  ls|cat|head|tail|find|grep|wc|echo|jq|date|pwd|which|type)
    echo '{"hookSpecificOutput":{"hookEventName":"PermissionRequest","decision":{"behavior":"allow"}}}'
    exit 0
    ;;
  git|npm|pnpm|yarn|node|python3|pip|rg|fd|bat|eza)
    echo '{"hookSpecificOutput":{"hookEventName":"PermissionRequest","decision":{"behavior":"allow"}}}'
    exit 0
    ;;
esac

# Unknown - let user decide
exit 0
```

## Fully Autonomous Operation Pattern

Combine PermissionRequest with other settings for zero-prompt operation:

```json
{
  "permissions": {
    "allow": [
      "Read",
      "Glob",
      "Grep",
      "Bash(git *)",
      "Bash(npm *)",
      "Bash(pnpm *)"
    ],
    "deny": [
      "Bash(rm -rf *)",
      "Bash(sudo *)"
    ]
  },
  "hooks": {
    "PermissionRequest": [
      {
        "matcher": ".*",
        "hooks": [
          {
            "type": "command",
            "command": "/home/neo/.claude/hooks/universal-permission.js"
          }
        ]
      }
    ]
  }
}
```

The flow:
1. **Permission rules** handle known patterns (allow/deny lists)
2. **PermissionRequest hook** catches anything that would prompt
3. **Hook decides** based on custom logic - no human in the loop

## Symbiont Impact

### Enabling True Autonomy

Before PermissionRequest, autonomous agents would stall at permission prompts. Even with aggressive allow lists, edge cases triggered dialogs. Now:

1. **Universal Interception**: Every permission dialog can be programmatically handled
2. **Context-Aware Decisions**: Hook has full access to tool input, session state, working directory
3. **Safe Fallbacks**: Unknown operations can still prompt (fail-safe)

### Use Cases

**1. Unattended Execution**
```
Agent runs overnight → hits permission prompt → hook auto-approves based on rules → continues working
```

**2. Tiered Approval**
```
Safe operations → auto-approve
Risky operations → deny with explanation
Unknown operations → queue for human review
```

**3. Audit Logging**
```
All permission requests flow through hook → log to ledger → then decide
```

### Integration with Permission Modes

| Mode | + PermissionRequest Hook |
|------|--------------------------|
| `default` | Hook handles most prompts; truly dangerous still blocked |
| `plan` | Hook auto-approves read operations; writes still reviewed |
| `acceptEdits` | Hook handles non-edit permissions |
| `bypassPermissions` | Hook still fires but usually unnecessary |

### Recommended Pattern

```javascript
// Universal permission handler for autonomous operation
const ALWAYS_ALLOW = [
  /^Read$/,
  /^Glob$/,
  /^Grep$/,
  /^mcp__.*__(get|list|read|search|retrieve)/,
];

const ALWAYS_DENY = [
  /Bash.*rm\s+-rf\s+[\/~]/,
  /Bash.*sudo/,
  /mcp__.*__delete/,
];

// Everything else: allow for autonomous mode, prompt for interactive
```

## Known Issues

1. **Race Condition**: Hook and dialog may fire in parallel. If hook takes >1-2 seconds, dialog may still appear momentarily (fixed in later versions)
2. **VSCode Extension**: PermissionRequest hook may not fire in VSCode extension on Windows
3. **Documentation Gap**: Input schema was initially missing from docs (fixed post-release)
