## Summary

v2.1.9 introduces **PreToolUse additionalContext** - the ability for hooks to inject context into Claude's conversation BEFORE a tool executes. This is a game-changer for context-aware operations, enabling dynamic context injection from external systems (like PostgreSQL) based on what Claude is about to do. Also adds `${CLAUDE_SESSION_ID}` for skills and `plansDirectory` for custom plan storage.

---

## What Changed

- **PreToolUse additionalContext**: Hooks can return `additionalContext` string that gets injected before tool execution
- **`${CLAUDE_SESSION_ID}`**: Skills can reference the current session ID for session-aware behavior
- **`plansDirectory` setting**: Store plan files in custom location
- **MCP `auto:N` threshold**: Configure how many MCP tools must match before auto-enabling
- **External editor (Ctrl+G)**: In AskUserQuestion "Other" input fields
- **Session URL attribution**: Commits and PRs from web sessions include session URL
- **Bug fixes**: Parallel tool call failures in long sessions, MCP reconnection hangs, Unicode JSON parsing

## Deep Dive: PreToolUse additionalContext

### The Problem It Solves

Before v2.1.9, hooks could:
- **Block** tool calls (exit code 2)
- **Allow/deny/ask** permissions
- **Modify** tool inputs via `updatedInput`

But there was **no way to inject context** into Claude's understanding before a tool ran. PostToolUse had `additionalContext`, but it only fired on success - if a command failed, Claude never got the context.

### How It Works

PreToolUse hooks can now return JSON with `additionalContext`:

```json
{
  "hookSpecificOutput": {
    "hookEventName": "PreToolUse",
    "permissionDecision": "allow",
    "additionalContext": "Context string injected into Claude's conversation"
  }
}
```

The `additionalContext` string is added to Claude's context **before** the tool executes. Claude sees this information regardless of whether the tool succeeds or fails.

### Key Characteristics

| Aspect | Behavior |
|--------|----------|
| **Timing** | Before tool execution |
| **Visibility** | Claude sees it, user sees in verbose mode |
| **Combined with** | Can combine with `permissionDecision` and `updatedInput` |
| **Multiple hooks** | If multiple hooks return additionalContext, they're concatenated |

### Practical Example: CLI Documentation Injection

```python
#!/usr/bin/env python3
import json
import sys
import os

input_data = json.load(sys.stdin)
tool_name = input_data.get("tool_name")
tool_input = input_data.get("tool_input", {})

if tool_name == "Bash":
    command = tool_input.get("command", "")

    # Detect first-time use of a CLI tool
    if command.startswith("saffi "):
        docs_path = os.path.expanduser("~/.local/share/saffi/docs.md")
        if os.path.exists(docs_path):
            with open(docs_path) as f:
                docs = f.read()

            output = {
                "hookSpecificOutput": {
                    "hookEventName": "PreToolUse",
                    "additionalContext": f"SAFFI DOCUMENTATION:\n{docs}"
                }
            }
            print(json.dumps(output))

sys.exit(0)
```

### Practical Example: Context from Database

```python
#!/usr/bin/env python3
import json
import sys
import psycopg2

input_data = json.load(sys.stdin)
tool_name = input_data.get("tool_name")
tool_input = input_data.get("tool_input", {})

if tool_name == "Read":
    file_path = tool_input.get("file_path", "")

    # If reading health-related files, inject relevant soul data
    if "health" in file_path or "meals" in file_path:
        conn = psycopg2.connect("postgresql://...")
        cur = conn.cursor()
        cur.execute("""
            SELECT content FROM soul_core
            WHERE category = 'health_context'
            ORDER BY updated_at DESC LIMIT 1
        """)
        row = cur.fetchone()

        if row:
            output = {
                "hookSpecificOutput": {
                    "hookEventName": "PreToolUse",
                    "additionalContext": f"HEALTH CONTEXT:\n{row[0]}"
                }
            }
            print(json.dumps(output))

sys.exit(0)
```

## Deep Dive: ${CLAUDE_SESSION_ID}

Skills can now reference the current session ID:

```yaml
---
name: log-activity
description: Log activity to Queen's Ledger
---
Log this activity to the Queen's Ledger with session ID: ${CLAUDE_SESSION_ID}
```

### Use Cases

1. **Session-keyed storage**: Store data in database keyed by session
2. **Cross-skill coordination**: Multiple skills can share state via session ID
3. **Audit logging**: Track which session performed which actions
4. **Queen's Ledger integration**: Log specialist activity with session identifier

## Symbiont Impact

### Critical for Context Priming

This is the missing piece for **intelligent context injection**. Before v2.1.9, vector priming happened at session start. Now it can happen **dynamically** based on what Claude is about to do.

### Use Cases for Symbiont

**1. Soul-Aware File Access**
```python
# When health-coach reads meal logs, inject recent sleep data
if specialist == "health-coach" and "meals" in file_path:
    inject(get_recent_sleep_data())
```

**2. Wound/Vow Pattern Detection**
```python
# When touching something related to a known wound, inject context
if file_touches_wound_pattern(file_path):
    inject(get_wound_context())
```

**3. Cross-Agent Awareness**
```python
# Before any operation, inject recent Queen's Ledger activity
inject(get_recent_ledger_entries())
```

**4. Relationship Context**
```python
# When reading communication files, inject contact context
if "emails" in file_path or "messages" in file_path:
    inject(get_contact_context(extract_contact(file_path)))
```

### Integration Pattern

```
┌─────────────────────────────────────────────────────────────┐
│                    PreToolUse Hook                          │
│                                                             │
│  1. Receive tool_name, tool_input from Claude               │
│  2. Query symbiont-db for relevant context                  │
│  3. Return additionalContext with injected knowledge        │
│  4. Claude sees context BEFORE tool executes                │
└─────────────────────────────────────────────────────────────┘
```

### Session ID for Specialist Tracking

Each specialist session gets a unique ID. Use `${CLAUDE_SESSION_ID}` in skills to:
- Log to Queen's Ledger which specialist is active
- Key session-specific data in PostgreSQL
- Coordinate between specialists via shared session awareness

## Configuration

### PreToolUse Hook Configuration

```json
{
  "hooks": {
    "PreToolUse": [
      {
        "matcher": "Read|Edit|Write",
        "hooks": [{
          "type": "command",
          "command": "python3 ~/.claude/hooks/context-injector.py",
          "timeout": 5
        }]
      }
    ]
  }
}
```

### plansDirectory Setting

```json
{
  "plansDirectory": "~/symbiont/plans/"
}
```

Store plan files in version-controlled location.
