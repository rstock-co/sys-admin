## Summary

v2.1.20 introduces **--add-dir CLAUDE.md loading** - the ability to load CLAUDE.md instructions from multiple directories via the `--add-dir` flag. This enables instruction layering for specialists without symlinks. Also adds PR review status indicators to the prompt footer and task deletion via TaskUpdate.

- **--add-dir CLAUDE.md Loading**: Load CLAUDE.md files from additional directories (requires env var)
- **PR Review Status**: Visual indicator in prompt footer showing PR state (approved, changes requested, pending, draft) with clickable link
- **Task Deletion**: TaskUpdate tool can now delete tasks with `status: "deleted"`
- **Vim Arrow History**: Arrow key history navigation in vim normal mode when cursor cannot move further
- **External Editor Help**: Ctrl+G shortcut added to help menu for discoverability
- **Session Compaction Fix**: Fixed issues preventing proper resume after compaction
- **Wide Character Rendering**: Fixed emoji and CJK character rendering artifacts
- **Background Agents Permission Prompt**: Now prompts for permissions before launching
- **Config Backup Rotation**: Timestamped and rotated (5 most recent kept)

## Deep Dive: --add-dir CLAUDE.md Loading

### The Problem It Solves

Before v2.1.20, sharing instructions across multiple Claude Code sessions required:
- **Symlinks**: Each specialist folder symlinks to `core/` for shared primitives
- **Copy/paste**: Duplicate instructions across folders
- **Parent directories**: Only upward traversal from cwd was supported

The `--add-dir` flag existed but only extended **file access**, not instruction loading.

### How It Works

With the environment variable enabled, CLAUDE.md and CLAUDE.local.md files in additional directories are loaded alongside project memories.

### Environment Variable (Required)

```bash
export CLAUDE_CODE_ADDITIONAL_DIRECTORIES_CLAUDE_MD=1
```

This is **opt-in** for backward compatibility. Without it, `--add-dir` only provides file access, not instruction loading.

### Configuration Methods

```bash
# CLI argument
claude --add-dir /path/to/shared/instructions

# In-session command
/add-dir /path/to/shared/instructions

# Settings (settings.json or settings.local.json)
{
  "additionalDirectories": ["/path/to/dir1", "/path/to/dir2"]
}
```

### Key Characteristics

| Aspect | Behavior |
|--------|----------|
| **Env var required** | `CLAUDE_CODE_ADDITIONAL_DIRECTORIES_CLAUDE_MD=1` |
| **File types loaded** | CLAUDE.md, CLAUDE.local.md |
| **Multiple directories** | Yes, all specified directories are scanned |
| **Order** | Additional dirs loaded after cwd chain |
| **Documentation** | Currently undocumented (GitHub Issue #21138) |

### Practical Example: Specialist Spawning

```bash
#!/bin/bash
# spawn-mycelium.sh v2

export CLAUDE_CODE_ADDITIONAL_DIRECTORIES_CLAUDE_MD=1

# Layer instructions: core → specialist
claude \
  --add-dir ~/symbiont/core/ \
  --cwd ~/symbiont/mycelium/$1
```

This replaces the symlink pattern:
```
# OLD: Symlinks in each folder
mycelium/health-coach/
├── CLAUDE.md
├── output-style.md
└── core/ → ../../core/  # Symlink

# NEW: --add-dir at launch
mycelium/health-coach/
├── CLAUDE.md
└── output-style.md
# core/ loaded via --add-dir flag
```

## Deep Dive: Task Deletion

### The Problem It Solves

The task system (v2.1.16) supported creating and completing tasks, but not removing them. Long sessions accumulated irrelevant tasks that cluttered the task list.

### How It Works

TaskUpdate tool now accepts `status: "deleted"`:

```json
{
  "taskId": "1",
  "status": "deleted"
}
```

### Task Lifecycle (Complete)

```
pending → in_progress → completed
                     ↘
                       deleted (permanent removal)
```

### Use Cases

1. **Session cleanup**: Remove exploration tasks that led nowhere
2. **Superseded tasks**: Delete tasks replaced by better approaches
3. **Cross-agent coordination**: Remove tasks another specialist handled
4. **Overnight daemon**: Clean up completed tasks during batch processing

## Symbiont Impact

### Critical: Instruction Layering Without Symlinks

This directly enables the SGTA architecture evolution:

**Current Pattern:**
- Each specialist folder has symlinks to `core/`
- Symlinks can break, are filesystem-specific
- Instruction sources not visible in launch command

**New Pattern:**
- Launch command explicitly shows instruction sources
- No filesystem dependencies (symlinks)
- SGTA factory can dynamically compose instruction sets
- Multiple instruction layers composable at runtime

### Integration with Autonomous Operation

For headless/autonomous spawning:

```bash
#!/bin/bash
# Autonomous specialist spawn with instruction layering

export CLAUDE_CODE_ADDITIONAL_DIRECTORIES_CLAUDE_MD=1
export CLAUDE_USE_SUBSCRIPTION="true"
unset ANTHROPIC_API_KEY

claude \
  --add-dir ~/symbiont/core/ \
  --cwd ~/symbiont/mycelium/$SPECIALIST \
  -p "Execute the task in your instructions"
```
