# Claude Code v2.1.0 - Deep Research

**Release Date:** January 7, 2026
**Tier:** Major
**Symbiont Relevance:** Critical
**Researched:** 2026-01-27

---

## Summary

v2.1.0 introduces **`context: fork`** for skills - the ability to run skills in isolated sub-agent contexts that don't pollute the parent conversation. This is a foundational feature for autonomous operation, enabling "fire and forget" workflows where utility tasks execute with full session awareness but return only a brief summary. Also ships skill hot-reload, the `agent` frontmatter field, wildcard tool permissions, and the `/teleport` command. A dense release with 1,096 commits.

---

## What Changed

- **`context: fork` for skills**: Skills can now run in isolated sub-agent contexts using frontmatter
- **`agent` frontmatter field**: Specify which agent type executes a forked skill (`Explore`, `Plan`, custom)
- **Skill hot-reload**: Skills in `~/.claude/skills` or `.claude/skills` activate immediately without restart
- **Hooks in skill frontmatter**: Add `PreToolUse`, `PostToolUse`, `Stop` hooks scoped to skill lifecycle
- **`user-invocable: false`**: Hide skills from `/` menu while keeping them available to Claude
- **Wildcard tool permissions**: Pre-approve command families like `Bash(npm *)` or `Bash(git * main)`
- **`/teleport` command**: Move sessions to claude.ai/code for web continuation
- **Shift+Enter newlines**: Works out-of-box in iTerm2, WezTerm, Ghostty, Kitty
- **Language setting**: Configure Claude to respond in specific languages
- **Security fix**: OAuth tokens and API keys no longer appear in debug logs
- **Skill progress display**: Real-time tool execution indicators while skills run

## Deep Dive: `context: fork`

### The Problem It Solves

Before v2.1.0, skills had two options for execution context:

1. **Inline (default)**: Skill runs in main conversation. Output stays in context. Every utility task (logging, note-taking, summaries) pollutes the conversation with irrelevant tokens.

2. **Subagent delegation**: Claude spawns a subagent. Clean isolation, but subagents start with a **clean slate** - they don't know what happened in the parent session.

**The gap:** Many utility tasks need full session awareness but shouldn't persist their output. A `/summary` skill needs to read the conversation to write notes, but its internal reasoning doesn't belong in the main thread.

### How `context: fork` Works

```yaml
---
name: session-summary
description: Summarize session and write to notes
context: fork
agent: Explore
---

Review this entire session and write a summary to ~/notes/sessions/${CLAUDE_SESSION_ID}.md
```

When this skill runs:

1. **Fork**: Creates a copy of the current conversation context
2. **Execute**: Skill runs with full session awareness in the fork
3. **Return**: Only a brief completion message returns to main context
4. **Discard**: Forked context is thrown away (results don't pollute main thread)

### Key Characteristics

| Aspect | Inline (default) | Subagent | `context: fork` |
|--------|------------------|----------|-----------------|
| Session history | Yes | No (clean slate) | Yes (forked copy) |
| Output persists | Yes | Summary only | Brief confirmation |
| Context pollution | High | None | None |
| Use case | Reference content | Delegated tasks | Utility workflows |

### The `agent` Field

When using `context: fork`, the `agent` field specifies which agent type executes:

```yaml
---
name: deep-research
description: Research a topic thoroughly
context: fork
agent: Explore
---
```

Available agent types:
- **`Explore`**: Read-only tools (Glob, Grep, Read) - optimized for codebase exploration
- **`Plan`**: Planning and analysis focus
- **`general-purpose`**: Full tool access (default if omitted)
- **Custom agents**: Any `.claude/agents/*.md` file you've defined

### Practical Example: Fire-and-Forget Logging

```yaml
---
name: log-activity
description: Log session activity to Queen's Ledger
context: fork
disable-model-invocation: true
allowed-tools: Bash(echo *), Write
---

Review this session and append a summary to ~/symbiont/ledger/${CLAUDE_SESSION_ID}.md

Include:
- What was accomplished
- Key decisions made
- Files modified
- Timestamp
```

This skill:
- Has full session context (knows what happened)
- Writes to external files
- Returns "Logged activity to ledger" to main context
- Discards all internal reasoning and file-reading output

### Practical Example: Research Without Pollution

```yaml
---
name: investigate
description: Deep-dive research on a topic
context: fork
agent: Explore
---

Thoroughly investigate $ARGUMENTS:

1. Search codebase with Glob and Grep
2. Read all relevant files
3. Analyze patterns and relationships
4. Return a concise summary with file references

Focus on actionable findings, not exhaustive lists.
```

The forked Explore agent can read dozens of files, grep thousands of lines, and produce a dense summary - all without flooding the main context with intermediate tool outputs.

## Deep Dive: Skill Hot-Reload

### Before v2.1.0

Creating or modifying a skill required restarting Claude Code. This friction discouraged experimentation.

### After v2.1.0

Skills in `~/.claude/skills/` or `.claude/skills/` activate immediately:

1. Create `~/.claude/skills/my-skill/SKILL.md`
2. It's immediately available as `/my-skill`
3. Edit the file, changes apply instantly

This enables:
- Rapid skill prototyping
- Live debugging of skill behavior
- Session-specific skill creation

## Deep Dive: Agent-Scoped Hooks

Skills can now define hooks that only run during that skill's lifecycle:

```yaml
---
name: safe-deploy
description: Deploy with verification
context: fork
hooks:
  PreToolUse:
    - matcher: Bash
      hooks:
        - type: command
          command: "echo 'Bash command: $TOOL_INPUT' >> /tmp/deploy.log"
  PostToolUse:
    - matcher: Bash
      hooks:
        - type: command
          command: "python3 ~/.claude/hooks/verify-deploy.py"
---

Deploy the application...
```

These hooks fire only when the skill is active, not during normal conversation. Perfect for:
- Audit logging for sensitive operations
- Post-execution verification
- Skill-specific guardrails

## Symbiont Impact

### Critical for Autonomous Operation

`context: fork` is the missing piece for autonomous workflows. Before this, every autonomous skill polluted the session context with its internal reasoning. Now specialists can:

1. **Execute complex workflows** without context overflow
2. **Maintain session awareness** for informed decisions
3. **Return clean summaries** that preserve context for future work

### Use Cases for Symbiont

**1. Session Logging**
```yaml
context: fork
```
Every specialist can log to Queen's Ledger without cluttering the session. The log captures full session context while the main thread stays focused.

**2. Background Research**
```yaml
context: fork
agent: Explore
```
When a specialist needs to research before acting, fork an Explore agent. It reads dozens of files, returns a summary, and discards the noise.

**3. Cross-Agent Handoff**
```yaml
context: fork
agent: custom-specialist
```
Spawn a forked specialist to handle a sub-task. It inherits session context, executes autonomously, returns results.

**4. Wound Pattern Detection**
```yaml
context: fork
```
Before touching sensitive areas, fork a sub-agent to analyze the situation. It can read soul data, check patterns, and return guidance without exposing internal reasoning.

### Integration Pattern

```
┌─────────────────────────────────────────────────────────────┐
│                    Main Conversation                         │
│                                                             │
│  User: "Deploy the health-coach changes"                    │
│                                                             │
│  Claude: Let me run the deploy skill...                     │
│          ┌─────────────────────────────────┐                │
│          │  FORKED CONTEXT                 │                │
│          │                                 │                │
│          │  - Full session awareness       │                │
│          │  - Reads 15 files               │                │
│          │  - Runs test suite              │                │
│          │  - Verifies deployment          │                │
│          │  - Logs to Queen's Ledger       │                │
│          │                                 │                │
│          │  (All this discarded)           │                │
│          └─────────────────────────────────┘                │
│                         │                                   │
│                         ▼                                   │
│  Claude: Deployed successfully. 3 files changed,            │
│          tests passing, logged to ledger.                   │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

### Basic Forked Skill

```yaml
---
name: my-skill

```

### Forked Skill with Custom Agent

```yaml
---
name: research-skill
description: Research something thoroughly
context: fork
agent: Explore
allowed-tools: Read, Grep, Glob
---

Research $ARGUMENTS and return findings...
```

### Forked Skill with Scoped Hooks

```yaml
---
name: monitored-skill
description: Skill with execution logging
context: fork
hooks:
  PreToolUse:
    - matcher: ".*"
      hooks:
        - type: command
          command: "echo '[$(date)] $TOOL_NAME' >> ~/.claude/skill-audit.log"
---

Instructions...
```

## Known Limitations

- **Skill tool invocation**: When a skill is invoked via the `Skill` tool (programmatically), `context: fork` and `agent:` fields are ignored - skill runs in main context
- **No streaming results**: Forked skills complete fully before returning; no partial progress visible in main thread
- **Context size**: Forked context inherits full parent conversation, so very long sessions may hit limits
