## Summary

v2.1.16 introduces a **native task management system** with dependency tracking, replacing the ad-hoc TodoWrite approach. This is the most significant new primitive for agent coordination since background agents were added. Tasks can declare `blockedBy`/`blocks` relationships, enabling complex multi-step workflows with automatic unblocking when dependencies resolve.

## Deep Dive: Native Task System

### The Four Task Tools

| Tool | Purpose | Key Parameters |
|------|---------|----------------|
| **TaskCreate** | Create new task | `subject`, `description`, `activeForm` |
| **TaskList** | View all tasks | None |
| **TaskGet** | Get full task details | `taskId` |
| **TaskUpdate** | Modify task | `taskId`, `status`, `owner`, `addBlockedBy`, `addBlocks` |

### TaskCreate Parameters

```json
{
  "subject": "Review authentication module",
  "description": "Review all files in app/services/auth/ for security vulnerabilities",
  "activeForm": "Reviewing auth module..."
}
```

| Field | Form | Example |
|-------|------|---------|
| `subject` | Imperative | "Implement X", "Add Y", "Fix Z" |
| `activeForm` | Present continuous | "Implementing X", "Adding Y", "Fixing Z" |
| `description` | Detailed requirements | Full context and acceptance criteria |

### TaskUpdate Parameters

```json
{
  "taskId": "2",
  "status": "in_progress",
  "owner": "security-reviewer",
  "addBlockedBy": ["1"],
  "addBlocks": ["3", "4"]
}
```

| Field | Purpose |
|-------|---------|
| `status` | `pending` → `in_progress` → `completed` |
| `owner` | Agent claiming the task |
| `addBlockedBy` | Tasks that must complete before this one can start |
| `addBlocks` | Tasks that cannot start until this one completes |

### How Dependency Tracking Works

1. **Create tasks** with sequential IDs (#1, #2, #3...)
2. **Set dependencies** via `addBlockedBy` or `addBlocks`
3. **Automatic unblocking** when blockers complete
4. **No polling required** - system handles state transitions

```
Task #1: Pull sleep data         [no dependencies]
Task #2: Analyze patterns        [blockedBy: #1]
Task #3: Generate briefing       [blockedBy: #2]
Task #4: Check urgent emails     [no dependencies]
Task #5: Compile final report    [blockedBy: #3, #4]
```

Task #2 auto-unblocks when #1 completes. Task #5 waits for BOTH #3 AND #4.

### Task Storage

Tasks persist at:
```
~/.claude/tasks/{sessionId}/*.json
```

**Critical limitation**: Tasks are **session-scoped**. Different sessions don't share task state. Tasks disappear when session ends.

### Task Visibility

TaskList returns:
- `id`: Unique identifier
- `subject`: Task title
- `status`: pending | in_progress | completed
- `owner`: Assigned agent (if any)
- `blockedBy`: List of blocking task IDs

A task is **available** when:
1. Status is `pending`
2. No owner assigned
3. `blockedBy` list is empty (all dependencies resolved)

## Multi-Agent Coordination Patterns

### Pattern 1: Sequential Pipeline

```
TaskCreate({ subject: "Research" })           // #1
TaskCreate({ subject: "Plan" })               // #2
TaskCreate({ subject: "Implement" })          // #3
TaskCreate({ subject: "Test" })               // #4
TaskCreate({ subject: "Document" })           // #5

TaskUpdate({ taskId: "2", addBlockedBy: ["1"] })
TaskUpdate({ taskId: "3", addBlockedBy: ["2"] })
TaskUpdate({ taskId: "4", addBlockedBy: ["3"] })
TaskUpdate({ taskId: "5", addBlockedBy: ["4"] })
```

### Pattern 2: Fan-Out / Fan-In

```
TaskCreate({ subject: "Analyze auth" })       // #1
TaskCreate({ subject: "Analyze API" })        // #2
TaskCreate({ subject: "Analyze DB" })         // #3
TaskCreate({ subject: "Synthesize findings" }) // #4

TaskUpdate({ taskId: "4", addBlockedBy: ["1", "2", "3"] })
```

Tasks #1-3 run in parallel. Task #4 waits for ALL to complete.

### Pattern 3: Worker Pool (Self-Organizing Swarm)

Multiple workers race to claim available tasks:

```python
# Worker loop:
while True:
    tasks = TaskList()
    available = [t for t in tasks
                 if t.status == "pending"
                 and not t.owner
                 and not t.blockedBy]
    if available:
        TaskUpdate({ taskId: available[0].id, owner: "my-id", status: "in_progress" })
        # Do work...
        TaskUpdate({ taskId: available[0].id, status: "completed" })
```

Natural load balancing - workers grab free work as it becomes available.

## Symbiont Impact

### Critical for Overnight Daemon

The task system enables structured overnight workflows:

```
Morning Briefing Pipeline:
├── Task 1: Pull Oura sleep data
├── Task 2: Analyze patterns (blockedBy: 1)
├── Task 3: Pull calendar for today
├── Task 4: Check urgent emails
├── Task 5: Generate briefing (blockedBy: 2, 3, 4)
└── Task 6: Send to inbox (blockedBy: 5)
```

**Before v2.1.16**: Single script, linear execution, no parallelism.
**After v2.1.16**: Parallel tasks (#1, #3, #4 run simultaneously), automatic dependency resolution.

### Cross-Specialist Coordination

Mycelia could create tasks that multiple specialists work on:

```
Mycelia creates:
├── Task: Health analysis (owner: health-coach)
├── Task: Financial review (owner: business-savant)
├── Task: Relationship check (owner: relationship-coach)
└── Task: Synthesize weekly review (blockedBy: all above)
```

**Limitation**: This requires all specialists to share a session OR a "hydration pattern" where task specs are stored externally and re-created in each session.

### The Hydration Pattern

Since tasks don't persist across sessions, use external storage:

```yaml
# ~/symbiont/task-specs/morning-briefing.yaml
tasks:
  - id: oura-pull
    subject: Pull Oura sleep data
    description: Fetch last night's sleep metrics
  - id: analyze
    subject: Analyze patterns
    blockedBy: [oura-pull]
    description: Look for trends and anomalies
  # ...
```

SessionStart hook reads spec, creates tasks via TaskCreate, sets dependencies via TaskUpdate.

### Limitations for Symbiont

1. **Session-scoped**: Tasks don't persist across sessions
2. **No cross-session sharing**: Independent sessions can't see each other's tasks
3. **Requires orchestration**: For multi-specialist work, need external coordination layer
4. **Storage location**: `~/.claude/tasks/{sessionId}/` - not in project folder

### Action Items

1. **Evaluate for overnight daemon**: Replace linear scripts with task-based workflows
2. **Design hydration pattern**: External task specs → SessionStart hook → TaskCreate
3. **Test dependency resolution**: Verify automatic unblocking works reliably
4. **Consider PostgreSQL bridge**: Store task state in Neon for cross-session visibility
