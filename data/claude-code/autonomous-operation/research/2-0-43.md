## Summary

v2.0.43 completes the multi-agent observability story with the **SubagentStart hook event**, allowing external systems to intercept subagent spawning. Combined with SubagentStop (added in v2.0.42), you can now track the full lifecycle of any specialist activation. The release also adds **permissionMode** for custom agents and a **skills frontmatter field** for auto-loading skills into subagents.

## What Changed

### New Features
- **SubagentStart hook event** - Fires when any subagent is spawned
- **permissionMode for custom agents** - Control permission behavior per agent
- **skills frontmatter field** - Declare skills to auto-load for subagents
- **tool_use_id field** - Added to PreToolUse and PostToolUse hook inputs

### Bug Fixes
- Fixed nested CLAUDE.md files not loading when @-mentioning files
- Fixed duplicate rendering of some messages in the UI
- Fixed visual flickers
- Fixed NotebookEdit inserting cells at incorrect positions when cell IDs matched `cell-N` pattern

## Deep Dive: SubagentStart Hook

### When It Fires

SubagentStart triggers **when a subagent is spawned** - before the subagent begins execution. This complements SubagentStop (v2.0.42) which fires when the subagent completes.

### Hook Input Schema

```json
{
  "session_id": "abc123",
  "transcript_path": "~/.claude/projects/.../abc123.jsonl",
  "cwd": "/Users/...",
  "permission_mode": "default",
  "hook_event_name": "SubagentStart",
  "agent_id": "agent-abc123",
  "agent_type": "Explore"
}
```

| Field | Description |
|-------|-------------|
| `session_id` | Main session identifier |
| `transcript_path` | Path to main session's transcript |
| `cwd` | Current working directory |
| `permission_mode` | Current permission mode |
| `hook_event_name` | Always "SubagentStart" |
| `agent_id` | Unique identifier for this subagent instance |
| `agent_type` | Agent name - built-in (`Bash`, `Explore`, `Plan`) or custom agent names |

### Configuration

Add to settings.json (project or user level):

```json
{
  "hooks": {
    "SubagentStart": [
      {
        "matcher": "*",
        "hooks": [
          {
            "type": "command",
            "command": "/path/to/log-specialist-activation.sh"
          }
        ]
      }
    ]
  }
}
```

### Matcher Patterns

Use matchers to target specific agent types:

```json
{
  "hooks": {
    "SubagentStart": [
      {
        "matcher": "health-coach",
        "hooks": [
          { "type": "command", "command": "./scripts/health-coach-activated.sh" }
        ]
      },
      {
        "matcher": "Explore",
        "hooks": [
          { "type": "command", "command": "./scripts/exploration-started.sh" }
        ]
      },
      {
        "matcher": "*",
        "hooks": [
          { "type": "command", "command": "./scripts/log-any-agent.sh" }
        ]
      }
    ]
  }
}
```

### Complete Lifecycle Tracking

With v2.0.42's SubagentStop and v2.0.43's SubagentStart, you can track the full lifecycle:

```json
{
  "hooks": {
    "SubagentStart": [
      {
        "hooks": [
          { "type": "command", "command": "./scripts/agent-started.sh" }
        ]
      }
    ],
    "SubagentStop": [
      {
        "hooks": [
          { "type": "command", "command": "./scripts/agent-stopped.sh" }
        ]
      }
    ]
  }
}
```

SubagentStop includes additional fields:
- `agent_transcript_path` - Path to subagent's own transcript (in `subagents/` folder)
- `stop_hook_active` - Whether a stop hook is already active

## Deep Dive: permissionMode for Custom Agents

### Available Permission Modes

| Mode | Behavior |
|------|----------|
| `default` | Standard permission checking with prompts |
| `acceptEdits` | Auto-accept file edits |
| `dontAsk` | Auto-deny permission prompts (explicitly allowed tools still work) |
| `bypassPermissions` | Skip ALL permission checks |
| `plan` | Plan mode (read-only exploration) |

### Usage in Agent Frontmatter

```yaml
---
name: autonomous-worker
description: Background worker that runs without prompts
tools: Read, Grep, Glob, Bash, Edit, Write
permissionMode: bypassPermissions
---

You are an autonomous worker. Complete tasks without asking for confirmation.
```

### Permission Inheritance

Important rules:
1. If parent session uses `bypassPermissions`, it **cannot be overridden** - all subagents inherit it
2. Otherwise, subagents can set their own permissionMode
3. Missing permissionMode defaults to parent's mode

### Security Considerations

From the docs: "When using `bypassPermissions`, all subagents inherit this mode and it cannot be overridden. Subagents may have different system prompts and less constrained behavior than your main agent. Enabling bypassPermissions grants them full, autonomous system access without any approval prompts."

## Deep Dive: skills Frontmatter Field

### Purpose

The `skills` field injects skill content directly into the subagent's context at startup. Unlike dynamically invoked skills, these are **preloaded** - the full content is immediately available.

### Usage

```yaml
---
name: api-developer
description: Implement API endpoints following team conventions
skills:
  - api-conventions
  - error-handling-patterns
---

Implement API endpoints. Follow the conventions and patterns from the preloaded skills.
```

### Key Points

1. **Full content injection** - Not just availability, the actual skill content is loaded
2. **No inheritance** - Subagents don't inherit skills from parent; must declare explicitly
3. **Startup loading** - Skills are injected when the subagent spawns, not on demand

### Difference from context: fork

| Approach | Who controls system prompt | Use case |
|----------|---------------------------|----------|
| `skills:` in agent | Agent controls, skill content injected | Agent needs specific knowledge |
| `context: fork` in skill | Skill controls, runs in specified agent | Skill defines the workflow |

## Symbiont Impact

### Critical: Logging to Queen's Ledger

SubagentStart enables intercepting specialist spawning for coordination and observability. This is essential for the Queen's Ledger pattern.

#### Example: Log Specialist Activations

Hook configuration:

```json
{
  "hooks": {
    "SubagentStart": [
      {
        "hooks": [
          {
            "type": "command",
            "command": "$CLAUDE_PROJECT_DIR/.claude/hooks/log-to-ledger.sh"
          }
        ]
      }
    ]
  }
}
```

Hook script (`log-to-ledger.sh`):

```bash
#!/bin/bash
# Log specialist activation to Queen's Ledger

INPUT=$(cat)
AGENT_TYPE=$(echo "$INPUT" | jq -r '.agent_type // "unknown"')
AGENT_ID=$(echo "$INPUT" | jq -r '.agent_id // "unknown"')
SESSION_ID=$(echo "$INPUT" | jq -r '.session_id // "unknown"')
TIMESTAMP=$(date -Iseconds)

# Log to ledger file
LEDGER="$HOME/symbiont/queen/ledger.jsonl"
echo "{\"event\":\"specialist_activated\",\"agent_type\":\"$AGENT_TYPE\",\"agent_id\":\"$AGENT_ID\",\"session_id\":\"$SESSION_ID\",\"timestamp\":\"$TIMESTAMP\"}" >> "$LEDGER"

exit 0
```

#### Example: Full Lifecycle Tracking

Track both activation and completion:

```bash
#!/bin/bash
# log-to-ledger.sh - handles both SubagentStart and SubagentStop

INPUT=$(cat)
EVENT=$(echo "$INPUT" | jq -r '.hook_event_name')
AGENT_TYPE=$(echo "$INPUT" | jq -r '.agent_type // "unknown"')
AGENT_ID=$(echo "$INPUT" | jq -r '.agent_id // "unknown"')
SESSION_ID=$(echo "$INPUT" | jq -r '.session_id // "unknown"')
TIMESTAMP=$(date -Iseconds)

LEDGER="$HOME/symbiont/queen/ledger.jsonl"

case "$EVENT" in
  "SubagentStart")
    echo "{\"event\":\"specialist_activated\",\"agent_type\":\"$AGENT_TYPE\",\"agent_id\":\"$AGENT_ID\",\"session_id\":\"$SESSION_ID\",\"timestamp\":\"$TIMESTAMP\"}" >> "$LEDGER"
    ;;
  "SubagentStop")
    TRANSCRIPT=$(echo "$INPUT" | jq -r '.agent_transcript_path // "unknown"')
    echo "{\"event\":\"specialist_completed\",\"agent_type\":\"$AGENT_TYPE\",\"agent_id\":\"$AGENT_ID\",\"session_id\":\"$SESSION_ID\",\"transcript\":\"$TRANSCRIPT\",\"timestamp\":\"$TIMESTAMP\"}" >> "$LEDGER"
    ;;
esac

exit 0
```

### Use Cases for Symbiont

1. **Observability** - Know which specialists are active across all sessions
2. **Coordination** - Queen can route tasks based on active specialists
3. **Debugging** - Full audit trail of specialist activations
4. **Metrics** - Track specialist usage patterns over time
5. **Rate limiting** - Prevent too many concurrent specialists

### Combining Features

A fully autonomous specialist with preloaded skills and logging:

```yaml
---
name: health-coach
description: Health and wellness specialist
permissionMode: bypassPermissions
skills:
  - biohacking-protocols
  - neo-health-history
---

You are Neo's health coach specialist...
```

With hooks configured at project level for logging.

## Known Limitations

1. **No output control** - SubagentStart hooks cannot block agent spawning (unlike PreToolUse)
2. **No transcript yet** - At SubagentStart, the agent hasn't written anything to its transcript
3. **Session-scoped** - Hooks only fire within the session that spawns the subagent
